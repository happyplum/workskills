# 文档同步

完成后的文档与记忆同步审计。将已验证的代码现实与所有上下文载体比对，检测漂移与失真，识别困在工具私有记忆中的团队应知知识。默认仅审计；修复需显式批准。

**核心原则：** 已验证的代码现实优先于过时文档。关键事实声明不仅要「最新」，还须「真实」——经代码/配置/运行验证。工具私有记忆（serena 等，gitignore）中的团队应知知识是「可见性漂移」——审计发现后，仅经验证的知识才可提升到 git 跟踪的项目文档。

**本 skill 应在完成门控期间调用，当上下文同步是门控条件时。** 运行时随改随同步是实现代理自身的连续义务（见 AGENTS.md「上下文维护」）；本 skill 不替代它，只做完成时的回看审计。

## 模式

| 模式 | 触发 | 行为 |
|------|------|------|
| `audit` | 默认 | 仅生成漂移/同步报告 |
| `fix` | 用户明确说「修复/应用/更新/同步」 | 审计 + 应用已验证修复 + 执行记忆可见性同步 |

可选输入：`fix`、`range=<git范围>`（如 `origin/main...HEAD`）、`carrier=<类别>`。指定 carrier 时，其他载体列为「未审计——已过滤」。

## 工作流

### 1. 建立证据基线

读取根目录 + 最近本地 `AGENTS.md`、相关 `README.md` 产品规格章节。识别计划文件（`.omo/plans/`、`docs/plans/`、项目根 `*.plan.md`，或 AGENTS 指明的路径）。**枚举记忆工具目录**：`.serena/memories/`、`.cursor/`、`.github/copilot/`、`.cline*` 等存在则纳入。

> **显式声明**：读取 gitignore 的工具私有记忆目录，目的是识别应提升到团队可见载体的知识，不改变这些目录的 git 状态。

提取：必需载体、同步规则、完成门控规则、文档边界、记忆工具清单。

### 2. 确定实现面

```
git diff --stat <range>    # 文件级变更面
git diff --name-only <range>
```

识别触及的包/模块/服务/路由、新增/移除的端点/工作流/页面。优先审计变更面——禁止扫描整个仓库。若无法确定范围（新仓库、单次提交），对所有默认载体审计当前代码状态并注明限制。

### 3. 构建载体清单

优先使用项目中命名的载体。若无命名则使用默认值：

| 类别 | 文件 | 可见性 |
|-------|-------|-------|
| `readme` | 根目录 + 最近包的 `README.md` | 团队可见（git） |
| `agents` | 根目录 + 本地 `AGENTS.md` | 团队可见（git） |
| `docs` | `docs/**`（路线图、架构、缺口、API） | 团队可见（git） |
| `plans` | 计划/规格/进度文件 | 团队可见（git） |
| `memory-private` | `.serena/memories/*.md` 等**工具私有**记忆 | **仅装该工具者可见**（常 gitignore） |

> **关键**：`memory-private` 类载体对未安装对应工具的同事不可见。其中的团队应知知识必须同步到团队可见载体（见步骤 6）。

### 4. 先分类再判断

确定内容是**描述性的**（当前事实）、**规范性的**（规则）、**愿景性的**（路线图/提案），还是**历史性的**（变更日志）。仅描述性/规范性内容被现实推翻时才算漂移。

### 5. 漂移与真实性审计

对每个载体，将声明与已验证的代码/配置/计划比对。两个维度：

**漂移**（声明 vs 现实不一致）：
- 「未实现」/「未来」/「缺口」/「TODO」语句现已不成立
- 缺失的路由/端点/服务/工作流/配置面
- 过时的「已知缺口」/「限制」/「技术债」章节
- 工作流描述不再匹配执行实际
- 记忆条目编码了已推翻的结论

**真实性**（声明是否经得起验证）——按类型抽查：

| 声明类型 | 核对要求 |
|---|---|
| critical/high 漂移声明 | **必须**验证（读代码/配置/运行） |
| 架构/拓扑/接口/运行方式类声明 | **必须**验证 |
| 措辞/摘要/路线图类声明 | 可抽查 |
| 无法验证（已删除的历史功能等） | 标记并注明原因 |

标记：`verified`（已验证为真）/ `false`（已验证为假）/ `unverified`（**尝试验证但无法确认**，或非关键可豁免——非「未查」兜底）。`unverified` 须注明尝试过什么、为何无法确认。

### 6. 记忆可见性同步

复用步骤 5 对每条记忆的核对结果：**仅 `verified` 条目进入同步候选**；`false` 条目转入记忆失效处理；`unverified` 条目不自动提升。

**团队应知判据**：若一位**未安装该记忆工具**的同事，在缺少这条知识时会做出错误决策、重复踩坑或无法独立维护该模块，则属「团队应知」；否则保持私有。

**默认不同步**（保持私有）：实现过程笔记、调试发现、试验性记录、个人工作流偏好、会话特定速查——这些是个人辅助，不是团队应知的长期知识。只有经得起「同事缺少会出错」判据的才进入同步候选。

识别 `memory-private` 中通过判据的 `verified` 知识，检查团队可见载体是否已包含：

| 私有记忆内容 | 同步到 | 不同步（保持私有） |
|---|---|---|
| 规则/约定/禁令 | `AGENTS.md` | 会话特定速查 |
| 产品规格/架构/拓扑 | `README` 产品规格 或 `docs/` | 个人工作笔记 |
| 可复用指南/设计参照 | `docs/` | 临时调试记录 |
| 长期技术债/边界 | 产品规格 或 `docs/` | 已被代码/文档覆盖的冗余记忆 |

对每条通过判据的 `verified` 团队应知知识：团队可见载体缺失或过时 → 标记为**高优先级同步项**。已被代码推翻（步骤 5 标 `false`）→ 标记记忆为待失效。

> 待同步的记忆条目须先经步骤 5 真实性核对（仅 `verified`）；未验证者不进入步骤 9 写入队列。

### 7. 评级严重程度

| 优先级 | 定义 |
|----------|------------|
| **critical** | 错误规则/失真事实可能立即误导实现/运维 |
| **high** | 当前状态文档实质性不正确；**团队应知知识困在私有记忆** |
| **medium** | 不完整摘要、过时缺口列表、部分漂移、未抽查的真实性 |
| **low** | 措辞清理、轻微遗漏、外观问题 |

### 8. 生成同步报告

```
## 范围 — 模式、git 范围、触及区域、记忆工具清单
## 漂移摘要 — Critical/High/Medium/Low 数量
## 真实性核对 — verified/false 数量（unverified 不计总数，在发现项逐条列原因）
## 记忆可见性同步 — 私有记忆中需同步到项目文档的条目（内容→目标载体→当前状态）
## 发现 — 按文件：[优先级] 路径 | 声明类型 | 漂移/失真 | 证据 | 修复 | 状态
## 完成状态 — 文档/记忆已更新？同步已执行？需人工后续？
```

状态值：`audit-only` | `fix-ready` | `fixed` | `manual-decision-needed`

### 9. 应用修复（仅 fix 模式）

- **漂移修复**：仅更新有证据支持的声明，保留文档意图。AGENTS 一致性 → 两个文件都更新。
- **记忆可见性同步（先验证后提升）**：**仅当**记忆条目在步骤 5 中标记为 `verified` 时，才写入对应项目文档（按步骤 6 映射）。`unverified`/`false` 条目即使属于团队应知，也只标记为 `manual-decision-needed`，**不自动提升**——防止私有猜测污染权威文档。同步后在记忆中标注「已同步至 <路径>」。
- **记忆失效**：步骤 5 标 `false` 的私有记忆 → 更新或删除；标注推翻证据。
- **反向补全（文档→记忆）**：默认关闭。仅当满足全部三条时执行：(a) 该知识会被未来检索高频命中；(b) 当前缺失会导致检索落空；(c) 文档未充分覆盖、需记忆存一条索引。或用户在 fix 模式显式要求。

### 10. 复查

验证：已变更文档内部一致；无载体仍陈述旧事实或失真事实；私有记忆的团队应知知识已在 git 跟踪载体中落地；记忆与文档对同一事实的陈述一致（无矛盾镜像）。

## 载体检查清单

| 载体 | 检查项 |
|---------|-------|
| README/产品规格 | 已发布状态、路由/端点、服务表、命令/安装、缺口/技术债、架构 |
| AGENTS | 工作流事实、本地规则、载体列表、同步/完成规则 |
| docs/** | 路线图状态、缺口分析、需求基线、API 文档、架构、迁移 |
| 计划文件 | 复选框、证据、延迟条目、验收标准 |
| 私有记忆（serena 等） | 是否含团队应知知识但困在私有存储；结论是否当前；过时是否已失效；与项目文档是否矛盾 |

## 决策规则与边界情况

**优先级：** 用户指令 > 已验证代码现实 > 现有文档 > 代理推断。模糊时 → `manual-decision-needed`。

| 情况 | 操作 |
|-----------|--------|
| 私有记忆含团队应知知识，项目文档缺失 | high 级同步项；fix 模式下写入项目文档 |
| 私有记忆与项目文档对同一事实陈述矛盾 | 以已验证代码现实为准；更新错误一方 |
| 私有记忆已被代码推翻 | 标记记忆待失效；fix 模式下更新或删除 |
| 私有记忆未验证（步骤 5 标 unverified） | 标记 `manual-decision-needed`；不自动提升 |
| 愿景性文档写成了当前事实 | 标记——清晰标注则非漂移，但读起来像当前事实则算漂移 |
| 功能在 feature flag 后 | 「已实现，由 [flag] 门控」——而非「完全发布」 |
| 代码与计划不一致 | 代码现实优先用于文档 |
| 无记忆后端 | 报告操作为待处理 |
| 无法确定 git 范围 | 审计所有默认载体，注明限制 |

## 红旗

- 「测试通过了，文档大概没问题」
- 「README 就够了」
- 「记忆以后再同步」——知识正困在私有存储，同事看不到
- 「记忆里写了就行」——同事没装该工具，记忆对他们不可见
- 「差不多就行，真实性不用逐条验证」

这些意味着漂移/失真/知识孤岛正在被合理化。停下来审计。
