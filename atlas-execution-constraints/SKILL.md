# Atlas 执行约束

## 概述

Atlas 执行治理的权威来源。编码工作以两个外部 skill 为执行核心：

- **`subagent-driven-development`（SDD）**——单任务质量核心：每个 task 走 implementer → reviewer（spec compliance + code quality）→ fix 闭环，串行处理任务序列。
- **`dispatching-parallel-agents`（DPA）**——跨任务并行核心：2+ 独立任务（无共享状态、无顺序依赖）并行 dispatch，统一集成验证。

本 skill 不重复定义两者的执行细节，而是作为其上的治理约束层：确保执行前就绪、执行中双核心不被静默绕过、完成前有具体证据、并行 Wave 通过整体门控。核心原则：仅执行结构有效、可验证、有证据支撑的内容。

## 反例（不应触发）

| 输入模式 | 原因 |
|---|---|
| 「生成计划」 / "write a plan" | 计划阶段，非执行治理。 |
| 「只要解释流程，不执行」 | 纯解释无需执行治理。 |
| 「单文件微小文本修正」 | 琐碎编辑无需完整约束栈。 |

## 强制规则

| # | 规则 | 验证 |
|---|------|------|
| 1 | **核心就绪**：执行前确认 SDD 已加载；计划含 2+ 独立任务时 DPA 也必须加载。缺少必需核心则停止。该停止点发生在任何 `task()` 委托、路由判断或执行 TODO surface 展开之前 | 启动检查已执行，必需核心均确认加载 |
| 2 | **仅执行已审查计划**：缺失/矛盾/未就绪 → 停止并请求上游修复 | 计划已通过审查且可执行 |
| 3 | **编码核心不得静默绕过**：编码任务走 SDD 闭环（不得降级为顺序手写）；存在 2+ 独立任务时走 DPA 并行（不得退化为串行） | 任务派发可追溯到 SDD 闭环或 DPA 并行 dispatch |
| 4 | **并行 Wave 门控**：DPA 同一 wave 全部返回后，对每个返回给出明确裁决（通过/未通过），整组通过才进下一 wave；任一未通过先修复重审 | wave 整体裁决记录存在 |
| 5 | **证据先于完成**：无具体证据不标记任务完成；审查/核对关卡前提供产物证据；状态保持可恢复 | 完成声明附带具体证据 |
| 6 | **有界提级不扩范围**：任务边界/意图不变仅复杂度上升时允许有界 category 提级（规则见 `omo-gated-routing-rules`）；提级改变边界/意图/交付物 → 拒绝，请求修复/重规划 | 提级记录含边界不变理由 |

## 双核心职责边界

| 层 | 职责 | 归属 |
|---|---|---|
| 单任务执行质量 | implementer→reviewer(spec+quality)→fix 闭环、file handoff、progress ledger | **SDD** |
| 单任务内串行性 | 同一任务的 implementer→reviewer 串行（SDD 流程结构） | **SDD** |
| 跨任务并行调度 | 独立域识别、并行 dispatch、集成验证、冲突检查 | **DPA** |
| 路由经济性 | 直接 vs 委托、category 选择、提级理由、审查门控 | `omo-gated-routing-rules` |
| 执行治理约束 | 就绪验证、核心不被绕过、证据纪律、Wave 整体门控、三类任务质量门 | **本 skill** |

> **并行与串行的边界**：SDD 管「同一任务内部不并行」（implementer→reviewer 串行）；DPA 管「多个独立任务之间并行」。两者不冲突——独立任务并行 dispatch（DPA），每个任务内部仍走 SDD 串行闭环。

> **双核心作用域**：双核心约束编码类任务；审查与核对任务有独立质量门（见「三类任务质量门」），不强制走 SDD/DPA。规则 3 的「不得绕过」仅约束编码任务。

> **模型与 category 的分层关系**：`omo-gated-routing-rules` 决定 `task()` 的 category（派发前经济路由）；SDD 在其 implementer/reviewer 派发内细化模型层级。二者分层不冲突——category 为派发层，SDD 模型选择为子代理层。

## 三类任务质量门

| 任务类型 | 执行核心 | 完成标准 |
|---|---|---|
| **开发** | SDD 闭环 | reviewer 报告 spec ✅ + quality approved；Critical/Important findings 已修复并 re-review |
| **审查** | 独立 reviewer 子代理 | 明确裁决（通过/驳回）+ 具体发现；驳回则进入修复循环 |
| **核对** | 证据比对 | 声明的证据路径下产物存在且与改动一致；无证据即未核对 |

## 并行 Wave 门控

DPA 将独立任务编入同一并行 wave 后：

1. **dispatch 前**：确认 wave 内任务无共享写入资源、无顺序依赖（DPA 职责）
2. **全部返回后**（不轮询运行中的后台任务，等系统通知）：对每个返回给出明确裁决（通过/未通过），裁决维度含目标达成、改动一致性、测试通过、证据具体、无 wave 内冲突。核对机制（读 summary、查冲突、跑全量、抽查）见 DPA 的 Review and Integrate
3. **整组通过** → 标记该 wave 完成，进入下一 wave；**任一未通过** → 先修复重审，整组通过前不进下一 wave

> 这是「正确性优先于并行吞吐」的有意取舍：wave 内最慢或失败的任务会拖住已通过任务，但能防止错误跨 wave 级联。

> 交叉引用 `interrupted-subagent-recovery`：若 wave 执行被中断，恢复时按其中断恢复协议审计旧状态，不无脑重派。

## 执行就绪检查

执行任何任务前，验证：

- SDD 已加载；计划含 2+ 独立任务时 DPA 也已加载
- 计划已审查、结构有效：任务/依赖/验证引用一致，无悬空引用，无矛盾
- 每个任务声明了 `category`/`subagent_type`，或携带有理据的延迟路由标记
- 编码任务不省略 SDD；并行任务不省略 DPA（详见规则 3）

任何检查失败 → 停止并退回计划修复。

## 证据纪律

- 没有具体证据不得标记任务完成。
- 审查/核对关卡前，提供产物证据（测试输出、构建结果、diff、验证产物）。
- 状态保持可恢复：进度同时记录在 SDD 的 progress ledger 与可追踪的 TODO 中，不依赖会话记忆。

## 故障处理

- 必需核心缺失/矛盾 → 停止，退回修复。
- 验证失败 ≥2 次重跑或证据矛盾 → 触发提级（本 skill 执行纪律触发器）；提级时的 category 选择见 `omo-gated-routing-rules`。
- Wave 关卡证据缺失 → 保持状态不变，循环修复 + 重新验证。
- 任务或验证重新开启 → 重置受影响的 wave 状态直到恢复闭合。
- 提级会改变任务边界/意图/交付物 → 拒绝，请求修复/重规划。

## 审查驳回循环

当计划定义审查关卡时：(1) 附带证据运行审查者循环，(2) 若被驳回 → 按反馈修复，(3) 重新运行直到获得明确 OKAY。驳回不等于失败——是质量网的正常工作方式；跳过 re-review 才是违规。
